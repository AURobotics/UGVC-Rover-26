\section{Design Process}

The mechanical design process was the cornerstone of our rover’s development, aiming to create a robust, modular, and high-performance system capable of handling off-road environments. The design methodology followed an iterative cycle of concept generation, CAD modeling, simulation, manufacturing, and physical testing.

\vspace{-10pt}
\subsection{Requirement Definition}

We began by identifying the key mechanical requirements of the rover based on UGVC terrain profiles, mission tasks, and functional constraints:
\vspace{-5pt}

\begin{itemize}
    \setlength{\itemsep}{1.2pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item Must withstand impacts from rocky and sandy terrain
    \item Should allow for modular integration with electronic and sensing subsystems
    \item Must maintain traction and stability during sharp turns and climbs
    \item Should be manufacturable using accessible local resources
\end{itemize}

\vspace{-5pt}
These needs translated into functional targets for the frame, suspension, drivetrain, and wheels.

\vspace{-10pt}
\subsection{Frame Design}

The rover’s frame was designed using 2020 aluminum extrusion profiles. This approach offered numerous mechanical benefits:


\vspace{-5pt}
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item High strength-to-weight ratio
    \item Easy mounting for subsystems
    \item Simple reconfiguration during prototyping
    \item No need for welding, which helped avoid thermal deformation
\end{itemize}

We used angle brackets and flat nuts to assemble the profiles, making the frame rigid, modular, and easy to maintain or expand. The design allowed us to test multiple suspension positions and easily mount or relocate electronic components.

\subsection{Suspension and Damping System}

The mechanical team decided early in the design phase to implement a double wishbone suspension system — an advanced configuration typically found in high-performance vehicles. Its selection was based on its ability to:

\vspace{-5pt}
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item Maintain consistent tire contact with uneven ground
    \item Reduce body roll during turns
    \item Allow controlled wheel articulation and travel
\end{itemize}

The suspension arms were made from Aluminum 5083, laser-cut for dimensional accuracy, then manually bent using a bending machine. This material was chosen for its excellent strength, corrosion resistance, and workability. Each control arm was carefully designed in CAD to account for expected loads and the dynamic motion range of the wheels.

To complete the system, we custom-designed and implemented dampers and springs. These were tailored to the rover's weight and expected ground impacts. We used hand calculations and simulation data to estimate the required spring constant and damping coefficient, ensuring smooth ride quality and protection of internal components from shock loading.

\vspace{-5pt}
\subsection{Drivetrain and Wheels}

For propulsion, we used four brushed DC motors, one for each wheel. These were paired with 10-inch pneumatic off-road tires, which were chosen for their traction, shock absorption, and ability to handle soft ground.

To enhance the top speed, we designed a 2:1 gear reduction system, transferring power efficiently while balancing speed and torque. This system was custom-fabricated and aligned with the motor specifications to avoid overheating or overload.

\vspace{-5pt}
\subsection{Manufacturing Process}

All mechanical components were first modeled in SolidWorks. After thorough design reviews and interference checks, we exported 2D drawings for CNC laser cutting, ensuring high-precision profiles. The bending process was executed manually using bending dies and form jigs, requiring skill and careful alignment. Despite not having access to CNC bending, the team maintained high consistency through repeatability tests and dimensional inspections.

\subsection{Software Tools}

The rover's software development followed a modular, test-driven approach centered around ROS 2 (Foxy) running on a Raspberry Pi. Key tools used during development include:
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item \textbf{Gazebo + RViz:} For simulating sensors, testing navigation, and visual debugging.
    \item \textbf{MATLAB + Simulink:} For control system design, PID tuning, and motor modeling using the System Identification Toolbox.
    \item \textbf{OpenCV:} For all computer vision-based modules including lane, obstacle, and pothole detection.
    \item \textbf{Git + GitHub Projects:} For collaborative version control and agile task tracking.
    \item \textbf{Docker:} Used to standardize development environments across the software team.
    \item \textbf{Python (pySerial, NumPy, Matplotlib):} For serial communication, sensor parsing, and real-time plotting during testing.
    \item \textbf{Custom Embedded Debugger:} For interactive ROS $\leftrightarrow$ MCU communication over UART.
\end{itemize}

This toolchain enabled parallel development across subsystems, robust testing, and reproducibility of simulation and deployment environments.

\subsection{Budget Management}

The rover was developed within a tightly constrained budget, which required cost-effective yet reliable component selection. Budgeting principles included:
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item \textbf{Modular PCBs:} Separated logic, power, and motor boards for reusability and easier debugging.
    \item \textbf{Open-source software and free student licenses:} Leveraged ROS 2, Gazebo, OpenCV, and MATLAB student versions.
    \item \textbf{Component prioritization:} Focused spending on mission-critical components like IMU (BNO055), GPS (NEO-6M), and motor encoders.
    \item \textbf{Efficient power delivery:} Dual battery packs (2x12V in series) with DC-DC buck converters ensured safe power distribution without needing expensive power modules.
    \item \textbf{Local sourcing \& salvaging:} Some structural and passive components were recycled from past projects or obtained from local vendors at reduced cost.
    \item \textbf{Contingency Planning:} $\sim$10\% of the budget was reserved for hardware replacements due to risk of burnouts during testing.
\end{itemize}

Budget tracking was maintained using spreadsheets and versioned with Git to document iterations and track unit cost fluctuations.

\vspace{-5pt}
\subsection{Safety Measures}

Safety was a core design consideration both in hardware and software.
\vspace{-10pt}
\paragraph{Hardware Safety:}
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item \textbf{Mechanical E-Stop:} A hardwired emergency stop switch connected to the power board, cutting off motor power instantly.
    \item \textbf{Wireless Kill Switch:} Implemented using NRF24L01 module. If signal is received, it sets motor speed to zero.
    \item \textbf{Power Filtering:} Voltage regulators and filtering capacitors were integrated to protect sensitive electronics from ripple and noise.
    \item \textbf{Voltage Scaling for Monitoring:} ADCs only read scaled battery values using precision resistive dividers to avoid damage.
\end{itemize}
\vspace{-10pt}
\paragraph{Software Safety:}
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item \textbf{Watchdog Timers:} Ensured ESPs reset in case of communication stall or software hang.
    \item \textbf{Status Feedback Loop:} All critical systems (battery, IMU, navigation mode) are monitored and reported to the GUI. Warnings are issued if thresholds are exceeded or sensors are unhealthy.
    \item \textbf{Command Timeout:} Motors automatically stop if no valid command is received within a certain timeframe.
    \item \textbf{Mode Isolation:} Manual mode overrides all autonomous actions to ensure immediate user control in unsafe scenarios.
    \item \textbf{Threaded UART Handler:} Prevents communication bottlenecks that could lead to unsafe delays in command dispatch.
\end{itemize}

\section{Technical Innovation}

The mechanical and software branches introduced several technical innovations that enhanced the rover’s performance, modularity, and manufacturing efficiency.

\subsection{Mechanical Innovations}

\subsubsection*{a. Custom Damper and Spring Integration}
One of our most significant mechanical innovations was the design and integration of a custom spring-damper system. Unlike using off-the-shelf shock absorbers, we developed our own based on analytical modeling and empirical testing. This allowed us to:
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item Match the damping response to real terrain conditions
    \item Optimize ride quality by controlling suspension rebound
    \item Reduce the mechanical stress on the frame and electronics
\end{itemize}
This in-house development demonstrated our ability to integrate dynamic systems into compact, modular designs.

\subsubsection*{b. Precision Laser-Fabricated Wishbones}
Rather than rely on basic linkage systems, we implemented a fully custom wishbone suspension, manufactured using CNC laser cutting. Each arm was optimized in CAD for both geometry and material utilization, resulting in:
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item Improved stress distribution under dynamic loads
    \item Consistent geometry for smooth suspension operation
    \item Compatibility with damping components and bearings
\end{itemize}

\subsubsection*{c. Modular and Serviceable Frame}
The use of aluminum profile framing was not just for convenience — it was a strategic decision to allow for rapid prototyping, scalability, and serviceability. The mechanical team incorporated:
\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item Standardized hole patterns and bracket slots
    \item Sliding slots for component position adjustment
    \item Removable crossbars to access internal systems
\end{itemize}

\subsubsection*{d. Manual Bending Precision}
While many teams rely on CNC bending, we proved that manual bending, when done with well-designed jigs and attention to tolerancing, can produce competition-grade components. This is particularly valuable for teams with limited access to high-end fabrication tools.

\subsection{Software Innovations}

\paragraph{Custom Messaging System with Flow Control}
A modular, RTOS-compatible messaging system was developed with software flow control, hardware parity checks, and per-port mutex protection. It monitors byte timing and buffer usage, with an integrated debug mode that logs all traffic to an external terminal without interfering with core functions.

\paragraph{Dual-Layer Sensor Fusion}
We implemented a two-stage UKF architecture: a primary filter fuses IMU and wheel encoders for drift-resistant local estimation, while a secondary UKF integrates GPS via \texttt{navsat\_transform} for long-term accuracy. This hybrid design provides short-term responsiveness and global consistency.

\paragraph{Angle-Constrained Bézier Planner}
The Bézier path planner was extended to support heading constraints at start and end points, enabling smoother and curvature-limited paths. By planning over $(x, y, \theta)$ state vectors, the system supports tighter turns and better integration with controllers like Pure Pursuit.

\paragraph{ROS $\leftrightarrow$ Embedded Abstraction}
A custom, threaded UART protocol bridges ROS 2 and ESP32 microcontrollers. Messages are parsed asynchronously, and embedded nodes implement a minimal ROS-like structure. This enables scalable, modular communication resembling DDS pub/sub over serial.

\paragraph{Embedded Autotuning and System ID}
Motor controllers perform real-time system identification using encoder logs, with MATLAB used to model transfer functions and autotune PID gains. The tuned parameters are flashed to the MCU, reducing manual effort and improving control stability.

\paragraph{Interactive Debug Console}
An embedded console supports real-time telemetry, command input, and live reconfiguration of controller parameters. A debug policy ensures safe activation of print hooks without interrupting real-time execution, supporting in-field diagnostics and tuning.

\paragraph{Health Monitor with GUI Feedback}
A self-check system continuously monitors battery voltage, IMU calibration, and control modes. Alerts are pushed live to the GUI, offering real-time insight and fault warnings similar to automotive CAN systems.

\section{Mechanical Design}

Our primary objective was to develop a lightweight, stable, and high-speed rover that complies with competition constraints, thereby maximizing its chances of success. To achieve this, we applied various mechanical design principles, including vibration isolation techniques, to optimize the suspension system.

To ensure a suspension that was both durable and flexible, we selected spring steel as the material of choice. Leveraging the principles of mechanical vibration, we calculated the required damping coefficients for effective vibration control. While we were limited to the dampers available in the local market, we successfully identified and acquired the best-performing options within those constraints.

When selecting the wheels, we considered several factors, including traction, maneuverability, acceleration, and additional damping characteristics. Given the competition requirements, we determined that pneumatic (air-filled) wheels were the most suitable. Their superior ground traction and built-in shock absorption offered significant performance benefits for our rover.

A detailed analysis of the forces acting on the rover's structural links and components led us to choose 5083-grade aluminium sheet metal as the primary material. Aluminium offers a significantly lower weight compared to alternatives like steel, which is crucial for a 50 kg rover aiming to optimize acceleration, maneuverability, and energy efficiency—especially when powered by batteries or solar energy.

Beyond weight advantages, aluminium provides an excellent strength-to-weight ratio and is relatively easy to machine, weld, and form—allowing us to fabricate a complex yet efficient design. Its high recyclability and cost-effectiveness further supported our choice, especially when compared to other high-performance materials such as titanium or advanced composites.

For manufacturing, we employed laser cutting as the preferred method for shaping aluminium plates. Laser cutting offers high precision and minimal material waste, enabling us to maintain tight tolerances. Since the process uses a focused laser beam without physical contact, it eliminates tool wear, resulting in cleaner cuts and smoother surfaces—often with little to no post-processing required. Additionally, the reduced tooling and setup costs made it an economical solution for producing custom parts from our 6 mm aluminium sheets.

To simplify assembly, we primarily used screws and nuts to fasten components together. In areas requiring additional strength and reliability, we incorporated welding, providing robust joints where necessary without complicating the overall construction process.

\subsection{Frame Design}

The rover frame was designed in compliance with the UGVC Competition dimensional constraints, which specify a length between 90 cm and 200 cm, a width between 60 cm and 120 cm, and a maximum height of 120 cm excluding the mechanical emergency stop tower. The structure is fabricated using lightweight aluminum profile extrusions to ensure high strength-to-weight ratio and ease of assembly. The modular nature of the aluminum profiles facilitates structural adaptability and integration of subsystems such as suspension, electronics, and payload. The design prioritizes rigidity and accessibility while maintaining compliance with competition limits to ensure smooth transportation and operational efficiency.

The SolidWorks model features a symmetric and balanced geometry optimized for both stability and load distribution, with mounting provisions integrated directly into the design to accommodate subsystems such as motors, electronics enclosures, suspension arms, and control modules. Finite Element Analysis (FEA) tools within SolidWorks were used to validate the structural integrity of the frame under anticipated mechanical loads during operation, ensuring minimal deflection and sufficient safety factors. Additionally, the modular nature of the aluminum profiles allows for quick modifications and maintenance, making the design both robust and adaptable. This SolidWorks-based approach not only ensures precision in fabrication but also streamlines the integration of mechanical and electrical systems within the competition constraints.

\section{Electronic and Power Design}

Our electrical system consists mainly of three PCBs: the Power Distribution Unit, the Main Control Unit, and the Motor Control Board. Our objectives were to create an electrical system that is highly efficient, yet simple and easily maintained.

\vspace{-10pt}
\subsection{Main Control Unit}

The main board for the UGVC competition was designed based on the ESP microcontroller, integrating essential components required for navigation, communication, and control. The ESP microcontroller offers Wi-Fi and Bluetooth connectivity, making it ideal for wireless communication in embedded systems. It is also known for its high efficiency and integration compared to traditional microcontrollers. Its low power consumption and solid processing power make it suitable for real-time robotics applications.

MOSFETs were used to switch the laser module on and off. The GPS module provides the robot with location and positioning data, while the IMU module offers orientation and motion sensing. Pin headers were included to provide interfaces for different communication protocols such as UART, I\textsuperscript{2}C, and SPI. Power management was handled using voltage regulators and filtering capacitors to ensure stable power delivery to all components. The PCB was designed with compact routing and noise isolation in mind. The ESP microcontroller was placed centrally on the board to minimize trace length and enhance signal integrity. The board was also designed to be easily connected to other robot systems using standard pin headers.

\subsection{Power Board}

To provide the necessary input voltage for the power board, we used two 12V battery packs connected in series, resulting in a 24V main power source. This setup not only aligns with the input requirements of our buck converters but also ensures adequate voltage headroom for efficient regulation. Supplying 24V to the power board allows us to regulate and distribute power reliably to both the motors and other subsystems without direct high-load draw from the batteries. This approach simplifies power management while maximizing battery utilization and system stability.

Our power board is designed to efficiently distribute power to all components in the system using two buck converters. These converters step down the primary 24V input to 12V and 5V outputs. The 12V output powers medium-voltage components such as motor controllers and certain sensors, while the 5V output supports low-voltage devices like microcontrollers and communication modules. This stepped-down power distribution ensures voltage stability and protects sensitive electronics, while optimizing energy efficiency across the rover’s subsystems.

\section{Software Design}
The rover’s software architecture is centered around a ROS 2 framework deployed on a Raspberry Pi, which acts as the main compute unit. It interfaces with two ESP32 microcontrollers over UART, one dedicated to motor control and wheel odometry, and another responsible for sensors such as the IMU, GPS. The ROS 2 ecosystem handles tasks like localization, sensor integration, motion control, and communication with a remote GUI. This architecture decouples computation and low-level control, ensuring real-time responsiveness, efficient data flow between hardware components and the software stack,  and modularity for debugging, extension, and upgrade. 



\subsection{Core Functionalities}

\subsubsection{Lane Detection}
Accurate and reliable lane detection is a fundamental component of the autonomous navigation system. In this project, we implemented a robust lane detection module based on the Hough Transform, which is a widely used technique in computer vision for detecting geometric shapes such as lines.

The process begins by capturing real-time images from the onboard camera system. These images are then processed through several stages to enhance lane visibility and reduce noise:

\begin{enumerate}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item \textbf{Grayscale Conversion:} The input image is first converted into grayscale to simplify subsequent processing steps.
    \item \textbf{Noise Reduction:} A Gaussian Blur filter is applied to smooth the image and suppress high-frequency noise that may lead to false detections.

\item \textbf{Edge Detection:} The Canny Edge Detection algorithm is employed to identify sharp intensity gradients, which typically correspond to lane boundaries.

\item \textbf{
Region of Interest Masking:} A predefined polygonal mask is applied to focus the detection only on the lower portion of the image where lanes are expected to appear, thus reducing irrelevant detections.

\item \textbf{Hough Transform Application:} The processed edge map is subjected to the Probabilistic Hough Line Transform. This algorithm identifies straight lines by transforming points in the image space to a parameter space and finding intersections that represent lines.

\item \textbf{Line Filtering and Classification:} Detected lines are filtered based on their slope and position to isolate valid lane markings. Left and right lanes are classified accordingly.
\end{enumerate}
\vspace{-10pt}
\paragraph{\textbf{Mathematical Foundation}}
The Hough Transform maps points (x,y) in image space to the parameter space defined by the line equation:

\vspace{-15pt}
\[
\rho = x \cdot \cos\theta + y \cdot \sin\theta
\]

\text{Where:} \\
\(\rho\) is the perpendicular distance from the origin to the line. \\
\(\theta\) is the angle of the line with respect to the x-axis.

By discretizing this space and accumulating votes for each \((\rho, \theta)\) pair, the most prominent lines are identified based on peak accumulator values.
\vspace{-10pt}
\paragraph{\textbf{Performance Consideration}}

\begin{itemize}
    \setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
    \item The algorithm performs reliably on asphalt tracks with clearly marked white borders, as defined in the competition rules.
    
    \item It maintains detection accuracy across varying track widths and curvatures, ensuring smooth navigation.

    \item The system can operate in real time, meeting the vehicle’s speed and response constraints.


\end{itemize}

\paragraph{\textbf{Limitations and Mitigations}}


In cases of poor lighting or occlusions, detection performance may degrade. To address this, adaptive thresholding and dynamic region of interest adjustments were incorporated.

\subsubsection{Pothole Detection}
The hole detection system is a real-time computer vision application designed to identify circular features in a video stream, such as mechanical holes. It is implemented using OpenCV and operates through a structured image processing pipeline.

The system begins by capturing live video frames from a camera. Each frame is converted to grayscale to simplify analysis, then processed with Gaussian blur to reduce noise. Canny edge detection is used to highlight the boundaries of objects in the image.

Contours are extracted from the edge-detected image and filtered according to their geometric properties. Specifically, the system evaluates the area, circularity, and radius of each contour to determine whether it resembles a valid circular shape. This helps eliminate false positives caused by noise or non-circular objects.

When a valid circle is detected, the system overlays visual markers on the video feed, including a boundary circle, a center point, and the measured radius. A clear status message indicates whether a circle is currently detected.

In scenarios where the camera is not perpendicular to the surface containing the hole, perspective distortion may cause circular shapes to appear elliptical. To compensate for this, a perspective correction method can be applied. This involves either calibrating the camera in advance using known patterns (e.g., a checkerboard grid) or estimating a homography transformation based on reference points in the scene. Once corrected, the circularity analysis becomes more reliable even under oblique viewing angles.

Key parameters such as the blur strength, edge detection thresholds, and geometric limits can be adjusted to suit different environments or use cases. The system runs continuously and exits cleanly when prompted, releasing the camera and closing display windows

\subsubsection{Obstacle Avoidance}

 An obstacle detection system is implemented using computer vision techniques with OpenCV. The primary objective is to process camera or image input and identify obstacles within the robot’s environment.

Canny edge detection is applied to emphasize the boundaries of objects present in the scene. To focus the analysis on the most relevant area for navigation, a polygonal mask is used to isolate the central region of the image, effectively eliminating irrelevant edges and noise from the periphery.

The algorithm systematically scans each column of the region of interest from bottom to top, identifying the first edge pixel in each column and storing these points in an array. Green lines are drawn between adjacent edge points and from each point to the bottom of the frame, providing a visual representation of the free space available for movement. Additionally, contours are extracted from the cropped edge image and highlighted in red, further delineating the detected obstacles and their boundaries.

By processing visual input in this manner, the system enables the detection of obstacles and the visualization of navigable free space, forming a critical component for autonomous navigation.

 The system begins by receiving a goal location, expressed as 2D coordinates (x, y), which can be provided manually by the user or automatically generated through an external interface prior to the round. The waypoint navigation package calculates a direct geometric path - typically a straight line or smooth arc - from the robot’s current position to the goal. The generated path is published as a list of intermediate waypoints or a continuous trajectory.

To track the planned path, the robot uses a Pure Pursuit controller implemented in another ROS package. This controller works by looking ahead to a target point on the path and adjusting the robot’s heading and velocity to “pursue” it, thus enabling smooth and adaptive motion. The controller calculates the necessary linear and angular velocity commands in real time, based on the curvature and heading angle to the lookahead point. These velocity commands are published to the standard \texttt{/cmd\_vel} topic, which interfaces with the robot’s motor driver node or embedded microcontroller to actuate the wheels.

\subsubsection{Motor Control}

\textbf{Introduction} \\
Quadrature encoders generate two out-of-phase signals, and each edge (rising and falling) on both signals is counted, resulting in four counts per cycle (4x decoding). This provides high-resolution tracking of movement. Direction is determined by the phase relationship between the signals - if one signal leads the other, it indicates forward or reverse rotation. This entire process is handled automatically by a dedicated hardware peripheral. The encoder output is fed into a PID controller with clamping to prevent integral windup. Each motor is equipped with its own encoder and an individually tuned PID controller for precise, independent control.

\textbf{Controller Design} \\
We needed to design a speed controller for every wheel, it was so important as it would allow us to navigate the robot throughout the field. For designing the controller first we had to model each motor independently using the system identification toolbox.

The motors we were using fall under the category of DC motors which are characterized by their first order response for a step input. So we connected all of the four motors to the rover and made sure all the encoders were recording data for the rotations of each motor. Then a step input was applied to all the motors at the same time and the rotations of each motor were recorded and sampled at time $T_s$, then after time $T$ the motors were stopped, and the reading for every encoder was extracted from the EEPROM of the MCU. We used these data to find the exact transfer function of each motor using the system identification toolbox.

After estimating the transfer function it's time to build the controller. We used a PID controller for its easy implementation, intuitive constants tuning, and suitability for the project. We used a PID controller where all terms (P, I, D) were needed. We used an integrator to eliminate the steady-state error caused by the nature of the system, and a derivative controller to decrease the overshoot due to the large $K_p$ value we needed for the fast response.

PID is a closed-loop feedback controller, where it always tries to follow a setpoint defined by the pilot or the Navigation controller. In our case, the set point is a reference speed in meters per second we need the motor to follow exactly to ensure that the navigation is working fine. The closed-loop action works by constantly trying to eliminate the error between the measured value and the reference value, and the constants in the controller determine how fast and strong the controller brings the output to the desired speed. Tuning the constants will be mentioned in the section below.

\subsubsection{Vehicle Monitoring}

The rover’s system provides real-time feedback on the rover’s operational status, including battery voltage, navigation mode, and sensor health, and communicates this information to the user via the GUI node.

\textbf{Battery Level Monitoring} \\
The battery level is measured using a voltage divider circuit that scales the main 24V power supply down to a safe range for the ESP32's analog input (0–3.3V). The ESP32 reads this scaled voltage using its ADC, and the raw analog value is converted back to the actual battery voltage by multiplying it by a pre-calibrated constant derived from the resistor ratio in the divider. This provides a real-time estimate of the battery level, which is packaged and published as part of a custom Status message. Low voltage thresholds can be set in software to trigger warnings or fallback behavior (switching to manual mode or limiting speed).

\textbf{Navigation Mode Monitoring} \\
The current operational mode, either Manual or Autonomous, is tracked based on commands received from the GUI. This mode status is maintained within the ESP Node and published periodically as part of the Status message, ensuring the GUI reflects the rover’s active state.

\textbf{IMU and Sensor Health} \\
The BNO055 IMU offers internal self-check diagnostics and status registers that report calibration states and sensor faults (e.g., accelerometer, gyroscope, magnetometer calibration status). These values can be read via I2C and forwarded through the ESP Node. Monitoring this data helps ensure that the IMU is providing reliable orientation information, and alerts are sent to the GUI if a sensor is uncalibrated or failing.

All collected data are bundled into a structured ROS message and published to a dedicated topic \texttt{/vehicle\_status}. The GUI Node subscribes to this topic and displays the relevant data to the user in real-time. This monitoring system ensures that the operator is continuously informed of the rover’s internal state and can take corrective action promptly if needed.

\subsubsection{Sensor Fusion}

To achieve reliable and accurate localization across both short-range and long-range scenarios, the rover uses a multi-layered sensor fusion approach based on the \texttt{robot\_localization} package in ROS 2. Two instances of the Unscented Kalman Filter (UKF) node are launched, each tailored for a different type of navigation.

\textbf{Primary UKF Node:} \\
Fuses continuous high-frequency sensor data from the BNO055 IMU and wheel encoders. This fusion produces precise short-term odometry optimized for short-distance waypoint navigation, and is published on the \texttt{/odometry/filtered} topic. The high update rate ensures responsive motion control with minimal latency. The fusion minimizes drift and provides smooth pose estimates during continuous movement.

\textbf{Secondary UKF Node:} \\
Integrates the filtered odometry output from the primary UKF with GPS position data from the NEO-6M v2 module via the \texttt{navsat\_transform} node. This creates a drift-resistant long-term odometry solution suitable for long-distance navigation and GPS coordinate-based waypoint following. The GPS integration corrects accumulated drift from dead reckoning, ensuring accurate global positioning over extended operation periods. This fused estimate is crucial for long-distance autonomous navigation and waypoint tracking. This is shown in the figure below.

In addition to the filtered odometry outputs, the system also publishes unfiltered odometry data to dedicated topics (\texttt{/odometry/unfiltered}). These represent the raw position estimates from individual sensors before fusion. This separation allows for easier debugging, sensor fault detection, and fallback logic in case one sensor stream becomes unreliable.

To improve the accuracy of IMU data, a zeroing and calibration routine is executed at startup.

\subsection{System Integration}

The rover’s software system is designed using ROS 2 running on a Raspberry Pi, which acts as the central processing unit. All major functionalities, autonomous navigation, sensor data handling, motor control, and communication with the user interface, are implemented as ROS 2 nodes that communicate via topics and actions.


At the core of the system is the ESP Node, which handles bidirectional UART communication between the Raspberry Pi and two ESP32 microcontrollers. The ESP Node uses the pySerial library. It spins up a dedicated thread per ESP device to handle serial reads and writes concurrently, ensuring non-blocking communication. Binary messages received over UART are parsed and routed into ROS topics as follows: 


\begin{itemize}
\setlength{\itemsep}{1pt}
    \setlength{\parskip}{0pt}
    \setlength{\parsep}{0pt}
  \item IMU data published to \texttt{/imu} topic
  \item GPS published to \texttt{/odometry/gps} topic
  \item Wheel velocities published to \texttt{/wheel\_vel} topic
  \item Fused pose published to \texttt{/odometry/filtered} topic
  \item Motion commands published to \texttt{/cmd\_vel} topic
\end{itemize}

While control commands are packaged and sent back to the ESPs, It also involves software flow control to achieve the highest communication rate. This design cleanly abstracts the hardware layer from the core logic and ensures timely, structured data exchange with the rest of the ROS graph.

The \textbf{Localization Node} uses the \texttt{robot\_localization} package to fuse GPS, IMU, and wheel odometry, publishing a refined pose on the \texttt{/odometry/filtered} topic. This output is then consumed by the \textbf{Navigation Node}, which plans a path to a given waypoint and publishes motion commands to \texttt{/cmd\_vel}.

The \textbf{GUI Node} interfaces with the user’s interface over Wi-Fi (IP-based). It sends high-level commands (joystick inputs for manual mode and waypoints for autonomous mode) and receives status updates from the ESP Node, such as battery levels and current mode (Manual / Autonomous).

The camera feed is processed by dedicated \textbf{Image Processing}, \textbf{Object Detection}, and \textbf{Lane Detection} nodes for autonomous navigation, while simultaneously being streamed to the remote console via the GUI Node over Wi-Fi for monitoring and manual control purposes.

All these nodes work together in a modular and asynchronous fashion. This architecture allows the system to compute an accurate and real-time estimate of the rover’s position and orientation.
